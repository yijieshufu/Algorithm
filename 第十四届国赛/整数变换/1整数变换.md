
## 1. 题目核心分析

- **题意**：给定一个正整数 $n$，不断执行 $n \leftarrow n - \text{数位之和}(n)$，直到 $n=0$，求操作次数。
    
- **数据范围**：$n \le 10^9$（十亿）。
    
- **核心矛盾**：
    
    - 操作次数非常多。对于 $10^9$，每次减去几十，大约需要 **2000多万次** 循环。
        
    - Python 的运行效率相对较低，如果在循环中频繁进行耗时操作（如字符串转换），必定超时 (TLE)。
        

## 2. 解题思路演进

### 第一阶段：暴力模拟（30-60分）

最直观的写法是每次循环都用 `str(n)` 将数字转为字符串，再遍历字符求和。

- **缺点**：`str(n)` 会涉及内存分配和类型转换。在 Python 中，这是一个相对昂贵的操作。在 2000 万次的量级下，累积耗时会超过 1-2 秒，导致超时。
    

### 第二阶段：预处理/打表法（100分 AC）

为了通过 $10^9$ 的数据，必须将单次循环的时间压缩到极致。我们使用 **“空间换时间”** 的策略。

1. **预计算（打表）**：
    
    - 我们发现计算数位和是重复性的工作。
        
    - 我们可以提前算好 $0 \sim 1,000,000$ 范围内所有数字的数位和，存入一个数组（列表）。
        
    - 查表的时间复杂度是 $O(1)$，远快于实时计算。
        
2. **大数拆分**：
    
    - 对于超过 $1,000,000$ 的数（例如 $n = 987654321$），不需要开那么大的数组（内存会爆）。
        
    - 利用数学性质：$S(987654321) = S(987) + S(654321)$。
        
    - 我们将 $n$ 拆分为 `high` (高位部分) 和 `low` (低位部分)，分别查表相加。
        

## 3. 最终 AC 代码（详解版）


```Python
import sys

# ---------------------------------------------------------
# 优化核心：预处理 (Pre-computation)
# ---------------------------------------------------------
# 定义打表范围为 100万。
# 这样占用的内存很小（约几MB），但能覆盖绝大多数计算。
MAX_PRE = 1000000
digit_sums = [0] * MAX_PRE

# 初始化表：利用动态规划思想填充
# i 的数位和 = (i // 10) 的数位和 + (i % 10)
# 例如：sum(123) = sum(12) + 3
# 这样不仅生成快，而且完全避免了 str() 转换  提前计算好每位数之和
for i in range(MAX_PRE):
    digit_sums[i] = digit_sums[i // 10] + (i % 10)

def solve():
    # -----------------------------------------------------
    # 输入处理
    # -----------------------------------------------------
    input_data = sys.stdin.read().split()
    if not input_data:
        return
    n = int(input_data[0])
    
    ans = 0
    
    # -----------------------------------------------------
    # 主逻辑：模拟过程
    # -----------------------------------------------------
    while n > 0:
        # 获取当前数字的数位和
        if n < MAX_PRE:
            # 情况1：数字在表内，直接 O(1) 读取
            sub = digit_sums[n]
        else:
            # 情况2：数字超出表范围，拆分为高位和低位
            # 例如 n = 12345678, MAX_PRE = 1000000
            # high = 12, low = 345678
            # 数位和 = table[12] + table[345678]
            high = n // MAX_PRE
            low = n % MAX_PRE
            sub = digit_sums[high] + digit_sums[low]
        
        # 更新数值
        n -= sub
        # 步数加一
        ans += 1
        
    print(ans)

if __name__ == "__main__":
    solve()
```

## 4. 复杂度与性能总结

| **指标**          | **暴力版 (String)**         | **优化版 (打表)**             |
| --------------- | ------------------------ | ------------------------ |
| **单次操作复杂度**     | $O(\log_{10} n)$ (带较大常数) | **$O(1)$** (纯数组索引)       |
| **总时间复杂度**      | $O(N \log N)$            | **$O(N)$** (近似线性)        |
| **空间复杂度**       | $O(1)$                   | **$O(M)$** (M为表的大小)      |
| **适用编译器**       | 甚至可能超时                   | **PyPy3 (推荐)** / Python3 |
| **是否通过 $10^9$** | 否                        | **是**                    |

### 关键笔记

1. **Python 慢在哪里？** 在极其密集的循环中，`str()` 类型转换、对象创建是性能杀手。
    
2. **数位和的数学拆分**：$Sum(A \times 10^k + B) = Sum(A) + Sum(B)$。这是能将大数拆分成小数查表的理论基础。
    
3. **竞赛技巧**：遇到 Python 超时，优先考虑 **PyPy3** 提交；如果逻辑正确但依然超时，考虑**打表**或**减少循环内的对象创建**。
